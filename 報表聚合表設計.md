# 報表聚合表設計

## 目標

在**不改既有報表欄位格式**的前提下，加速：

- `VD 車道報表`
- `統計分析`
- `CSV / PDF 匯出`

做法不是改前端格式，而是把後端查詢從：

- 直接掃 `traffic_events` / `congestion_samples` / `lpr_records`

改成：

- 優先查**實體聚合表**
- 明細頁才查 raw table

---

## 設計原則

1. 報表格式欄位固定，不改 response shape。
2. 原始事件表保留，供追查與歷史頁使用。
3. 報表類查詢走聚合表，不直接掃大表。
4. 依顆粒度切不同聚合表：
   - `每分鐘 -> agg_1m`
   - `每5分鐘 -> agg_5m`
   - `每小時 -> agg_1h`
5. 採增量聚合，不做全表重算。

---

## 整體資料流

```text
traffic_events
congestion_samples
lpr_records
        ↓
每分鐘增量聚合
        ↓
traffic_report_agg_1m
congestion_report_agg_1m
lpr_report_agg_1h
        ↓
由 1m 再往上滾動
        ↓
traffic_report_agg_5m
traffic_report_agg_1h
congestion_report_agg_5m
congestion_report_agg_1h
        ↓
報表 API / 匯出 API
```

## 原始資料表

### 1. `traffic_events`

用途：

- 歷史追查

## 實體聚合表

## A. `traffic_report_agg_1m`

用途：

- VD 報表 `每分鐘`
- 其他高顆粒統計來源

主鍵建議：

- `bucket_start`
- `camera_id`
- `direction`
- `lane_no`

欄位建議：

```text
id
bucket_start              -- UTC, 該分鐘起始時間
bucket_size               -- 固定 '1m'
camera_id
road_name
direction
lane_no
total_flow
avg_speed
max_speed
avg_occupancy
small_vehicle_flow
large_vehicle_flow
max_queue_length_m
avg_queue_length_m
queue_duration_sec
event_count
updated_at
```

說明：

- `total_flow`：該分鐘事件數
- `avg_speed`：有效速度平均
- `avg_occupancy`：該分鐘事件 occupancy 平均
- `max_queue_length_m` / `avg_queue_length_m`：來自同時間桶內壅塞聚合
- `queue_duration_sec`：該時間桶內排隊成立秒數

---

## 聚合公式

## 1. VD / 車流

### 總流量

```text
total_flow = 該 bucket 內 traffic_events 筆數
```

### 平均速度

```text
avg_speed = 該 bucket 內有效 speed_kmh 平均
```

### 平均占量

```text
avg_occupancy = 該 bucket 內 occupancy 平均
```

### 小車 / 大車流量

建議先分兩類：

- 小車：`car`, `motorcycle`
- 大車：`bus`, `truck`

```text
small_vehicle_flow = 小車事件數
large_vehicle_flow = 大車事件數
```

### 最大 / 平均估計排隊長度

```text
max_queue_length_m = 同 bucket 的壅塞樣本最大值
avg_queue_length_m = 同 bucket 的壅塞樣本平均值
```

### 排隊持續時間

```text
queue_duration_sec = 同 bucket 內 queue_active 成立秒數總和
```

---

## 2. 壅塞

### 估計排隊長度

目前公式：

```text
排隊長度 ≈ Σ(各車等效長度 + 安全間距)
```

建議等效長度：

- `motorcycle = 2 m`
- `car = 6 m`
- `bus = 12 m`
- `truck = 12 m`
- `safety_gap = 1.5 m`

### 排隊持續時間

當 `queue_active = true` 時，依 `sample_interval_sec` 累加：

```text
queue_duration_sec += sample_interval_sec
```

## 聚合更新策略

## 方案：增量聚合

不做全表重算，只聚合新增資料。

### Step 1. 建一張聚合 job 狀態表

例如：

```text
aggregation_job_state
- job_name
- last_processed_at
- updated_at
```

### Step 2. 每分鐘執行一次

```text
讀取 last_processed_at
→ 抓出之後的新 traffic_events / congestion_samples / lpr_records
→ 算出受影響 bucket
→ upsert 到 agg_1m
→ 再由 agg_1m 滾動到 agg_5m / agg_1h
→ 更新 last_processed_at
```

### Step 3. 使用 upsert

避免重複寫入：

```text
INSERT ... ON CONFLICT (...) DO UPDATE
```

## API 查詢切換策略

## 1. VD 報表

既有欄位不變，只改資料來源：

```text
每分鐘 -> traffic_report_agg_1m
每5分鐘 -> traffic_report_agg_5m
每小時 -> traffic_report_agg_1h
```

輸出欄位維持：

- 設備編號
- 道路
- 時間
- 方向
- 車道數
- 總流量
- 平均速度
- 平均占量
- 最大估計排隊長度
- 平均估計排隊長度
- 排隊持續時間

## 2. 統計分析

總覽 KPI 與趨勢改查聚合表：

- 流量
- 平均速度
- 最大速度
- 占量
- 排隊長度

## 3. 匯出 CSV / PDF

沿用相同聚合查詢結果，不額外重算。

---

## fallback 策略

若聚合表尚未有資料：

1. 小時間範圍允許 fallback raw table
2. 大時間範圍禁止掃 raw table
3. 明確回傳：
   - `aggregation_pending`
   - 或 `資料尚在彙總中`

避免前端誤以為是 `0`。

## 結論

本專案不建議只做 `view`。

正確做法是：

- **raw table 保留明細**
- **實體聚合表負責加速**
- **API 維持原欄位格式不變**

也就是：

```text
不改報表格式
只改後端資料來源
```
